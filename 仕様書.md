# ランダム生成問題 × 自動採点システム 仕様書

対象読者: 「学生ごとに数値・条件が異なる問題を大量に作りたい」「手書き解答をスキャンして自動採点したい」教員。
本書は、Claude Code のようなコーディングAIアシスタントにそのまま渡して実装を依頼できることを目的とした発注書です。
科目は問いません（材料力学・電気回路・化学量論計算・統計学の検定問題など、数値解答のある演習であれば応用できます）。

---

## 1. 解決したい課題

- 同じ問題文をそのまま毎年・毎クラス使い回すと、模範解答が出回り演習・小テストの効果が薄れる。
- かといって教員が毎回手作業で数値を変えた問題を人数分作るのは非現実的。
- 大人数の記述式・計算問題を毎週手採点するのは負担が大きい。

**本システムの核心は次の2点だけ**です。それ以外（GUI、PDF美化、成績配布など）はすべて付随機能です。

1. **問題の"構造"はプログラムで固定し、数値・拘束条件・荷重や現象の種類などの"パラメータ"だけを乱数で決める。**
   これにより「解き方は同じだが答えは各人・各回で本質的に異なる」問題を無限に生成できる。
   さらに、パラメータを決めた乱数と同じロジックで正答も自動計算されるため、**模範解答を人力で作る必要がない**。
2. **手書き解答用紙をスキャンし、画像認識（AI）で数値を読み取って、自動生成した正答と照合し採点する。**
   採点は「完全一致のみ正解」ではなく、桁・符号・有効数字の近さに応じた部分点ロジックを持たせるのが実用上重要。

この2つが揃うことで、「出題→実施→採点→返却」の全工程を教員1人でも大量にさばけるようになる。

## 2. システム全体のパイプライン

```
[1] 問題定義              科目の知識をコードで表現する部分（教員が書く／AIに書いてもらう部分）
        │   パラメータ範囲・拘束条件・出題パターンを定義
        ▼
[2] 問題インスタンス生成    乱数シード → 具体的な数値・条件を持つ問題オブジェクト
        │   同じシードなら何度生成しても同一の問題・同一の正答になる（再現性）
        ▼
[3] 用紙PDF生成            問題文・図・解答欄・位置合わせマーカーを含むPDFを出力
        │
        ▼
[4] 実施・回収             印刷して配布 → 学生が手書き → スキャン/撮影で画像化
        │
        ▼
[5] 画像の正規化           マーカー検出 → 射影変換（ホモグラフィ）で回転・歪みを補正
        │
        ▼
[6] 解答欄の切り出し        解答欄1マスずつを個別画像として抽出
        │
        ▼
[7] 文字認識（AI）         数字・符号などをボックスごとに分類 → 予測値の一覧
        │
        ▼
[8] 人によるレビュー・補正  低信頼度・誤読の疑いがある箇所を人間が確認・修正（表形式で編集）
        │
        ▼
[9] 自動採点               [2]で自動生成した正答と、[7-8]の読み取り結果を照合し配点
        │
        ▼
[10] 集計・返却            個人結果PDF、クラス統計、返却物の配布
```

[1][2] が「科目固有の知識」、[3]〜[10] が「科目に依存しない共通エンジン」。
**この分離が最も重要な設計方針**です。新しい科目を追加するときは [1][2] だけを書き直せばよく、[3]以降は使い回せます。

## 3. 各工程の詳細仕様

### [1][2] 問題定義とインスタンス生成

- 問題は「パラメータクラスの組み合わせ」として定義する。例:
  - 数値パラメータ（範囲 `(最小値, 最大値, 刻み幅)` を指定。例: 荷重 1〜10 kN を 1 kN 刻み）
  - カテゴリパラメータ（複数の型・材料・条件からリストで選ぶ。例: 断面形状 = 円形/矩形/中空円形、材料 = 鋼/アルミ/銅）
  - 拘束条件パラメータ（支持条件・境界条件など、位置や種類も乱数で決める）
- **1つの問題インスタンスは整数ID（シード値）だけで一意に決まる。**
  - 出題時: `シード = 学生の出席番号` や `シード = 実施日(YYYYMMDD)` など、後から再現・追跡できる値を使う。
  - 乱数生成の直前に必ず `seed(id)` を呼び、他のプロパティ生成の副作用で乱数系列がずれないようにする。
  - この「ID→問題→正答」が決定論的であることが、後述の**再採点・補正・再テストの全てを支える**。
- **正答は問題生成と同じコード・同じ乱数から計算する**（別途手計算しない）。可能なら `sympy` 等の数式処理ライブラリで
  シンボリックに解いて厳密値を出し、最後に丸める。これにより「問題文と模範解答の食い違い」というありがちな事故を防げる。
- 出題パターン（大問・小問構成、何を問うか）自体も、リストからインデックス選択や複数パターンの合成で増やせる。
- 遅延評価にしておく（アクセスされて初めて生成・計算する）と、GUIやノートブックから触るときに扱いやすい。

**AIに発注する際に伝えるべきこと**: この科目でランダマイズしたいパラメータの一覧（値の範囲・単位・刻み幅）、
問題の型（何問構成か、各問が独立か前問に依存するか）、正答の計算方法（教科書の公式・アルゴリズム）。

### [3] 用紙PDF生成

- 出題用（模範解答を見せる演習プリント）と、採点用（小テスト、マーカー・解答欄つき）の2種類を作れるようにする。
- 採点用PDFには以下が必須:
  - **位置合わせマーカー**（用紙四隅など4点以上に黒塗りの図形）。スキャン画像の回転・傾き・遠近歪みを補正するための基準点。
  - **氏名・出席番号欄**（採点結果を個人に紐づけるため）。
  - **解答欄**（後述の「固定桁数フォーマット」に合わせたマス目）。
- 使用ライブラリ例: PDF生成 `reportlab`、図形描画 `svgwrite`/`svglib`（数式や図形をベクター描画してPDFに埋め込む）。

### [4]〜[6] スキャン・正規化・切り出し

- 学生はスキャナーまたはスマホ撮影で解答用紙を画像化する（多少の回転・歪みが生じる前提でよい）。
- マーカー検出 → OpenCV でホモグラフィ変換 → 用紙全体を正規化された矩形画像に補正する。
  - マーカーは用紙の端に近いほど補正精度が上がる（内側に置くと外側の歪みが補正しきれない）。
- 正規化後は解答欄の座標があらかじめ分かっているため、各マスを機械的に切り出せる。

### [7] 文字認識（AI）

**ここが最も「科目非依存で使い回せる」パーツ**であり、後述する認識精度の議論も参照。

- 解答欄1マスに1文字（数字1桁、または符号1つ）だけを書かせる設計にすると、認識問題が
  「自由形式の手書き文字認識」ではなく「固定候補からの分類問題（0〜9 + 空欄、+/− + 空欄など）」に単純化でき、
  精度・速度・実装コストの面で大きく有利になる。**これは本システムの設計上の最重要ポイントの一つ。**
- 数値解答は固定桁数フォーマットに正規化するとよい。例: 符号1桁 + 有効数字3桁 + 指数2桁 = 合計6桁。
  自由記述だと「12000」と「1.2×10⁴」のような表記ゆれで誤答判定になりかねないが、
  フォーマットを固定すれば表記ゆれの問題自体が消える。
- 認識結果はキャッシュ（CSV等）に保存し、一度推論した画像は再実行時にスキップする（再採点の高速化）。
- 複数モデルによるアンサンブル（多数決）投票を行うと、単一モデルより誤読率を下げやすい。

### [8] 人によるレビュー・補正

- 認識結果は「予測値の一覧を表形式で見て、誤りがあればワンクリックで訂正できる」UIまたは
  スプレッドシートの列編集で補正できるようにする。
- 補正後は自動採点をやり直すだけでよく、画像を再認識する必要はない。
- 特定回だけに起きた問題（印刷ミス、特定シードの解答条件の誤りなど）は、問題定義本体を汚さず
  「補正フック」的な仕組み（該当シードのときだけ挙動を上書きするパッチ）で当てると、
  本体のロジックを追跡しやすいまま運用できる。

### [9] 自動採点ロジック

数値解答は「完全一致でなければ0点」ではなく、桁・符号・近さに応じた部分点を与える設計が実用的。
例（本システムで採用している段階的配点。数値は一例、科目に応じて調整可）:

| 条件 | 配点係数 |
|---|---|
| 四捨五入後に完全一致 | ×1.0 |
| 倍半分の範囲内（0.33〜3倍）かつ符号一致 | ×0.8 |
| 有効数字（仮数部）が一致 | ×0.8 |
| 符号のみ異なり仮数部が一致 | ×0.8 |
| 絶対値の仮数部のみ一致 | ×0.6 |
| 絶対値が倍半分の範囲内 | ×0.6 |
| それ以外 | ×0 |

- 正答が複数通りあり得る設問（対称性・複数解が正しい場合）には、正答候補のリストを持たせ、
  最も高いスコアになる候補で採点する。
- 大問（複数の小問からなる問題）は配点を小問数で均等割りするなど、配点ルールを問題定義側から渡せるようにする。

### [10] 集計・返却

- 個人ごとの採点結果（正誤・部分点・正答の書き込み）をPDF等で返却する。
- クラス全体の統計（平均・中央値・順位・過去回との推移）を出せると、教員が難易度調整の参考にできる。
- 出席番号の重複・欠番など、実施運用上のヒューマンエラーを自動検出できると事故防止になる。

## 4. 手書き数字・記号認識モデルの選択肢

自作CNN（数字0〜9・空欄の11クラス分類、符号+/−/空欄の3クラス分類、複数モデルの投票アンサンブル）で
**実運用精度 約99.8%** を達成しているケースがある。この数値の位置づけと代替案を整理する。

### この精度をどう評価するか

- 「解答欄1マスに1文字だけを書かせる」設計にした時点で、問題は事実上 MNIST 相当の
  孤立手書き数字分類問題になっている。この種のタスクの学術的な精度上限はアンサンブル・データ拡張を
  駆使してもおおむね 99.7〜99.9% 程度であり、**99.8% は実運用（きれいなMNISTデータセットではなく、
  スキャン起因のノイズや癖字を含む）としてはこの分野でほぼ天井に近い水準**。
  つまり「箱に1文字」という設計を維持する限り、モデルを差し替えても劇的な精度向上は期待しにくい。
- 精度を追加で詰めるなら、モデル刷新よりも次の方が費用対効果が高いことが多い:
  - 学習データへのデータ拡張（弾性変形・回転・太さ変化などで疑似的にデータを増やす）
  - アンサンブルのモデル数を増やす／多様性を上げる（既に投票方式なら効果が出やすい）
  - 低信頼度（票が割れた箱）だけを人間レビューに回す仕組みの強化（[8]参照。精度を100%に近づけるのは
    モデル単体よりもこの人間補正フローの設計の方が効く）

### 代替・補完候補（比較表）

| 選択肢 | 特徴 | 向いているケース |
|---|---|---|
| **自作CNN + アンサンブル投票（現行方式）** | 学習データを問題フォーマットに合わせて用意できれば最も高精度・低コスト・オフライン完結。学習の初期投資は必要。 | 解答フォーマットを固定桁数化できる／個人情報を外部送信したくない／継続的に運用しモデルを育てられる。 |
| **マルチモーダルLLM（例: Claude・GPT・Gemini等の画像入力）による低信頼度箱の裁定** | 文脈（単位・妥当な値の範囲・前後の設問）を使って曖昧な文字を判断できる。1件ずつAPI呼び出しになるため低速・従量課金。 | 全件処理ではなく、CNNの票が割れた／空欄か判別できない等の**難読ケースのみ**の二次判定に使うと費用対効果が良い。全件をLLMに投げるのはコスト・速度の点で非推奨。 |
| **クラウドDocument AI/OCR（Google Document AI、Azure AI Document Intelligence、AWS Textract等の手書き文字対応OCR）** | 自由記述の文章・数式混じりの手書きなど、固定桁数フォーマットに正規化できない解答形式に強い。学習データを用意しなくても使える。 | 解答が数式・文章・図など、そもそも「箱に1文字」に単純化できない科目・設問形式。個人情報を外部送信する契約・プライバシー面の確認が前提。 |
| **PaddleOCR / EasyOCR 等のオープンソースOCR** | ローカル実行可・無料。ただし孤立手書き数字1文字の分類に特化したものではなく、単純な数字ボックス認識では自作CNNに精度で劣ることが多い。 | 自由記述の解答（記述式の文章問題等）で、かつ外部送信を避けたい場合の叩き台。 |
| **TrOCR（Transformerベースの手書きOCR）** | 行単位の手書き文字列認識に強い。文字単位ではなく文章単位の認識に向く。 | 解答が短い文章・式変形の途中式など、単純な数字ボックスでは表現しにくい形式。 |

**結論**: 解答形式を「箱に1文字」に設計できる科目（数値解答が中心の理系科目全般）であれば、
自作CNN＋アンサンブル投票＋人間補正フローという現行方式が精度・コスト・プライバシーの面で引き続き最有力。
記述式や自由記述の答案が中心の科目でシステムを作る場合は、クラウドOCRやLLMによる読み取りを
第一候補として検討するのが良い（ただしその場合、採点ロジックは「文字の完全一致」ではなく
キーワード抽出・LLMによる意味的な採点など、全く別の設計が必要になる点に注意）。

## 5. AIに発注するときに整理しておくべき項目

新しい科目・環境向けにこのシステムを作ってもらうAIアシスタントに渡す際は、次を先に決めておくと手戻りが少ない。

1. **科目・問題形式**: 何を問う科目か。数値解答か、選択式か、記述式か、複数が混在するか。
2. **ランダマイズしたい要素の一覧**: 数値の範囲・単位・刻み幅、条件・パターンの種類とその数。
   （例: 材料力学なら「荷重の種類・大きさ・作用位置」「断面形状・寸法」「支持条件の種類・位置」「材質」）
3. **正答の計算方法**: 手計算の手順、使用する公式・アルゴリズム、シンボリック計算が必要か。
4. **解答フォーマット**: 「箱に1文字」方式にできるか（できるなら自作CNN方式が有利）、
   それとも自由記述にせざるを得ないか（その場合はクラウドOCR/LLM方式を検討）。
5. **採点の許容範囲**: 完全一致のみか、桁・符号・近似値に応じた部分点を与えるか。複数正答を許容するか。
6. **実施規模と頻度**: 何人×何回分を捌く必要があるか（自動化にかける初期投資の妥当性判断に影響）。
7. **データの取り扱い**: 学生の解答画像・個人情報を外部クラウドに送信してよいか（プライバシーポリシー・契約の確認）。
   外部送信不可なら自作モデル・オフライン処理が前提になる。
8. **利用環境**: スキャナーかスマホ撮影か、GPU等の計算資源の有無（モデル学習・推論に影響）。

## 6. 参考: 本仕様のベースになった実装（材料力学科目での実例）

- 問題定義: パラメータ・拘束条件・荷重種別をクラスとして定義し、`Question`/`BigQuestion` の
  サブクラスとして各週の問題セットを記述。乱数シードは問題ID（出席番号・実施日）で固定。
- 用紙生成: PDF（reportlab）＋ SVGベクター図（svgwrite/svglib）。マーカー・氏名欄・解答欄を自動配置。
- スキャン正規化: OpenCVによるマーカー検出＋ホモグラフィ変換。
- 解答フォーマット: 符号1桁＋仮数3桁＋指数2桁の6桁固定。
- 認識: 自作CNN（数字11クラス／符号3クラス）＋複数モデルの投票アンサンブル。認識結果はCSVにキャッシュし、
  手動修正列で補正可能。
- 採点: 完全一致・倍半分・仮数一致などの段階的部分点ロジック。
- 運用: GUIから科目・週を選び「演習PDF生成／小テストPDF生成／採点／採点結果配布」をボタン操作で実行。
  採点後にクラス統計・個人成績PDFを自動生成し配布する。

この実例では「材料力学」という科目固有の知識（梁のたわみ計算・応力計算等）が [1][2] にあたる部分に
埋め込まれているだけで、[3]以降の用紙生成・スキャン・認識・採点・配布のエンジンは科目に依存しない。
他科目に展開する際は、この [1][2] 部分（問題定義）を新規に書き起こすのが作業の中心になる。

## 7. 開発時の個人情報の取り扱い（重要）

本システムが扱う成績集計・解答スキャン画像・学生名簿などは、実運用ではすべて個人情報である。
コーディングAIアシスタント（Claude Code等）に開発・保守を依頼する際は、次を徹底すること。

- AIが実装・デバッグのために必要とするのは、データの**構造**（列名・フォーマット・ファイル配置・
  ディレクトリ構成）であり、現物のデータそのものではない。
- 実際の成績データ・解答スキャン画像・名簿ファイルなど、個人が特定できる現物データにAIが
  直接触れる（読み込む・開く・要約する・出力に含める）ことは絶対に避ける。
- 動作確認・テストには、構造だけを模したダミーデータ（架空の氏名・出席番号・数値）を用意して使う。
- 現物データでの確認がどうしても必要な場合は、AIに直接見せず、人間側で内容を確認したうえで
  必要な情報（件数・フォーマットの一致/不一致など）のみをAIに伝える。
